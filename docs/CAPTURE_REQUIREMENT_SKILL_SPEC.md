# מפרט Skill: `capture-requirement`

עודכן: 2026-07-15

> מפרט בלבד. ה־Skill עדיין אינו מותקן ואינו משנה קבצים. מקור הכללים המחייב היחיד הוא `CLAUDE.md`.

## 1. מטרת ה־Skill
להפוך דרישה חדשה של יניב להצעת זיכרון מדויקת, קצרה וניתנת לביקורת—בלי לשמור אותה אוטומטית, בלי ליצור מקור אמת נוסף ובלי לחשוף מידע רגיש.

ה־Skill פותר את הבעיות הבאות:
- דרישה חשובה נשארת רק בשיחה ונשכחת בהמשך.
- אותה דרישה נשמרת בכמה ניסוחים.
- הוראה חדשה אינה מחליפה הוראה קודמת.
- state זמני נכנס בטעות לכללים קבועים.
- כלל טקסטואלי נשמר למרות ש־test או guard מתאימים יותר.
- דרישה מפרויקט אחד מופעלת בטעות בפרויקט אחר.

## 2. עיקרון האבטחה
ה־Skill הראשון הוא **קריאה בלבד**.

הוא רשאי:
- לקבל את נוסח הדרישה.
- לקרוא קובצי כללים, דרישות, state והחלטות לא־רגישים.
- לחפש כפילויות, סתירות ומקורות אמת.
- להחזיר הצעת שינוי מובנית.

הוא אינו רשאי:
- לערוך או ליצור קובץ.
- להריץ פקודת shell בעלת השפעה.
- לבצע commit, push, merge או deploy.
- לשנות DB, settings, permissions, Skills או Hooks.
- לקרוא `.env`, credentials, transcripts, browser profiles או מידע אישי.
- להחליט לבדו שדרישה היא קבועה.

הכתיבה תתבצע רק לאחר אישור יניב, באמצעות תהליך `safe-change` נפרד.

## 3. הפעלה
ה־Skill יהיה ידני בלבד:

```text
/capture-requirement [הדרישה החדשה]
```

מאפיינים נדרשים בעת היישום:
- `disable-model-invocation: true`.
- אין auto-trigger.
- חסימת כלי עריכה, כתיבה, shell, network ו־delegation באמצעות `disallowed-tools` ו־permissions התואמים לגרסת Claude Code המותקנת.
- אין להסתמך על `allowed-tools` כמנגנון הגבלה; הוא עשוי רק לאשר כלים מראש.
- לפני יישום יש לאמת את שמות הכלים וה־frontmatter מול גרסת Claude Code המקומית.

## 4. קלט
### קלט חובה
- נוסח הדרישה המדויק של יניב.

### קלט אופציונלי
- project/repository.
- feature/path.
- האם ההוראה מחליפה הוראה קודמת.
- מקור חיצוני או קובץ שאליו הדרישה קשורה.

אין לבקש מידע אופציונלי כאשר ניתן להסיק אותו בבטחה מהריפו ומהשיחה.

## 5. סיווג הדרישה
ה־Skill חייב לבחור סוג אחד ראשי:

| סוג | משמעות | יעד טיפוסי |
|---|---|---|
| `global-preference` | העדפה שחוצה פרויקטים | זיכרון אישי פרטי |
| `project-rule` | דרך עבודה קבועה בפרויקט | `CLAUDE.md` או Rule תחומי |
| `product-contract` | התנהגות מוצר שאסור לשבור | requirements/schema/test |
| `architecture-decision` | בחירה עמידה עם נימוק | `DECISIONS.md` |
| `current-state` | מצב זמני, branch, blocker או צעד הבא | `CURRENT_STATE.md` |
| `open-task` | פעולה לביצוע | Issue/checklist |
| `evidence` | תוצאת בדיקה או מקור | evidence/report מתאים |
| `sensitive` | מידע אישי, סוד או runtime רגיש | אין לשמור בריפו ציבורי |
| `not-memory` | משפט חד־פעמי שאינו דורש שמירה | לא נשמר |

אם הסיווג אינו ודאי, ה־Skill מציג עד שתי אפשרויות ואינו כותב.

## 6. סדר החקירה
ה־Skill עובד לפי הסדר הבא, ועוצר כאשר יש די ראיות:

1. `CLAUDE.md` — כללים מחייבים.
2. `CURRENT_STATE.md` — מצב זמני.
3. מסמך הדרישות הקנוני של התחום, אם ידוע.
4. `DECISIONS.md` — החלטות עמידות.
5. Rules לפי path הרלוונטי.
6. tests, schemas או guards שיכולים כבר לאכוף את הדרישה.
7. חיפוש ממוקד בלבד במסמכים נוספים.

אין לקרוא את כל הריפו, את כל ההיסטוריה או את כל מסמכי המחקר כברירת מחדל.

## 7. בדיקת כפילות וסתירה
ה־Skill בודק ארבע רמות:

1. **זהה** — אותו ניסוח או אותה משמעות.
2. **חופף** — הדרישה החדשה היא מקרה פרטי של דרישה קיימת.
3. **מעדכן** — הדרישה מחדדת או מרחיבה כלל קיים.
4. **סותר** — לא ניתן לקיים את שתי הדרישות יחד.

לכל התאמה יש להציג:
- הקובץ והסעיף.
- הסוג.
- הסבר קצר על היחס.
- הפעולה המוצעת: no-op / merge / replace / supersede / clarify.

אין להשאיר דרישה ישנה וחדשה פעילות יחד כאשר החדשה מחליפה את הישנה.

## 8. בחירת יעד השמירה
ה־Skill בוחר את המקום הקטן והקנוני ביותר:

- כלל שחייב להיות בכל משימה בפרויקט → `CLAUDE.md`.
- כלל שחל רק על path/feature → Rule לפי path.
- חוזה דטרמיניסטי → test, schema, verifier או guard; טקסט קצר מפנה אליו.
- state → `CURRENT_STATE.md`.
- החלטה → `DECISIONS.md`.
- משימה → Issue/checklist.
- העדפה גלובלית או מידע אישי → מערכת פרטית, לא קלוטקורד ציבורי.
- משפט חד־פעמי → לא נשמר.

ה־Skill אינו מציע ליצור קובץ חדש כאשר יעד קנוני קיים מתאים.

## 9. פורמט הפלט
הפלט יהיה קצר ומובנה:

```text
סיווג: [type]
Scope: [global / repo / path / feature]
רגישות: [public-safe / private / prohibited]

נוסח קנוני מוצע:
[משפט אחד עד שלושה]

נמצא בזיכרון:
- [אין / קובץ + סעיף + יחס]

פעולה מוצעת:
[no-op / update / replace / supersede / save-as-state / create-task / do-not-store]

יעד:
[path או מערכת פרטית]

אכיפה מומלצת:
[none / test / verifier / schema / Rule / Permission / Hook / Skill]

נדרש אישור:
[הפעולה המדויקת שתבוצע לאחר אישור]
```

יעד אורך: עד 350 מילים, אלא אם נמצאה סתירה מורכבת.

## 10. כללי ניסוח קנוני
- משפט חיובי וברור כאשר אפשר.
- כלל אחד בכל סעיף.
- ללא סיפור היסטורי או הסבר ארוך.
- ללא “תמיד” אם קיימים חריגים.
- ללא נתונים זמניים או תאריך בתוך כלל קבוע.
- scope מפורש.
- תנאי הצלחה ניתן לבדיקה כאשר אפשר.
- אין שכפול של כלל גלובלי במסמך תחומי.

## 11. טיפול במידע רגיש
אם הדרישה כוללת מידע אישי או סוד:
- אין לצטט את הערך הרגיש בפלט.
- מסווגים `sensitive`.
- מציעים placeholder או תיאור schema בלבד.
- יעד השמירה הוא config פרטי, vault, DB או מערכת פרטית.
- אין commit לריפו ציבורי.

## 12. מתי להציע אכיפה במקום זיכרון נוסף
יש להציע אכיפה כאשר הדרישה:
- ניתנת לבדיקה דטרמיניסטית.
- כבר נשכחה או נשברה בעבר.
- בעלת סיכון גבוה.
- נוגעת ל־secrets, Git, DB, auth, RTL, A4 או חוזה נתונים.

סדר העדיפות:
1. test/schema/verifier.
2. Permission.
3. Rule לפי path.
4. Skill לתהליך רב־שלבי.
5. Hook רק כאשר חובה לחסום תמיד ואין פתרון פשוט יותר.

## 13. מבחני קבלה
ה־Skill לא ייחשב מוכן עד שיעבור לפחות את התרחישים הבאים:

| תרחיש | תוצאה צפויה |
|---|---|
| העדפה גלובלית חדשה | מציע יעד פרטי ואישור; לא כותב לקלוטקורד הציבורי |
| דרישה שכבר קיימת | `no-op`; לא יוצר כפילות |
| דרישה חדשה שמחליפה קודמת | `supersede`; מציג מה יוחלף |
| מידע זמני על branch או blocker | מסווג `current-state` |
| משימה חד־פעמית | מסווג `open-task` או `not-memory` |
| מספר טלפון, token או שם תלמיד | `sensitive`; אינו מצטט ואינו שומר |
| כלל UI שחל רק על path מסוים | מציע Rule תחומי, לא `CLAUDE.md` גלובלי |
| חוזה דטרמיניסטי | מציע test/verifier ולא רק טקסט |
| שני מסמכים סותרים | מציג conflict ואינו בוחר אוטומטית |
| ניסוח עמום | מבקש החלטה אחת ממוקדת בלבד |
| פרויקט קטן ללא צורך בזיכרון נוסף | `not-memory` או עדכון מינימלי |
| דרישה מפרויקט אחר | מזהה scope שגוי ואינו מעתיק |

## 14. מדדי הפיילוט
במהלך 10 דרישות אמיתיות מודדים:
- כמה דרישות סווגו נכון.
- כמה כפילויות נמנעו.
- כמה סתירות זוהו.
- כמה פעמים נדרש תיקון סיווג.
- כמה דרישות נשמרו במקום הנכון.
- כמה דרישות הפכו ל־test/guard במקום טקסט.
- זמן ממוצע מהבקשה להצעה.
- context/usage של ההפעלה.
- האם דרישה קריטית נשכחה במשימות הבאות.

## 15. תנאי הצלחה, עצירה ו־rollback
### הצלחה
- אפס כתיבה ללא אישור.
- אפס מידע רגיש בריפו ציבורי.
- אפס כפילויות חדשות בעשר דרישות הפיילוט.
- לפחות 90% סיווג נכון לאחר review.
- הפלט קצר וברור ליניב.

### עצירה
- ה־Skill מציע יעד שגוי שוב ושוב.
- הוא קורא יותר מדי קבצים או צורך context גבוה.
- הוא מצטט מידע רגיש.
- הוא יוצר כללים במקום tests מתאימים.
- יניב נדרש להסביר יותר במקום פחות.

### Rollback
- ה־Skill מותקן בפרויקט פיילוט אחד בלבד.
- ההתקנה נמצאת ב־PR אחד.
- הסרה היא revert של אותו PR.
- הצעות שלא אושרו אינן משנות שום קובץ.

## 16. שלבי היישום
1. השלמת Inventory מקומי.
2. אימות frontmatter ושמות הכלים בגרסה המותקנת.
3. יצירת Skill read-only בפרויקט פיילוט אחד.
4. הרצת מבחני הקבלה הסינתטיים.
5. פיילוט של 10 דרישות אמיתיות.
6. החלטה: keep / adjust / remove.
7. רק לאחר הצלחה: חיבור לתהליך `safe-change` המאושר.
