# מדריך יומי קצר ליניב — עבודה נכונה עם Claude Code

## לפני שמתחילים
1. פתח את Claude מתוך תיקיית הריפו הנכונה.
2. ודא שהשם והנתיב נכונים.
3. אל תפעיל Opus או agent teams אוטומטית.
4. תן משימה אחת בלבד בכל session.

## כך כותבים משימה
```text
משימה: [מה צריך לקרות].
היקף: [מה מותר לשנות].
לא לשנות: [מה מוגן].
הצלחה: [איך נדע שזה עובד].
בדיקות: [preview / build / live].
עבוד בענף ופתח PR; אל תמזג לפני דיווח.
```

## בחירת מודל
- שינוי קטן או קוד שוטף: Sonnet.
- debugging מורכב: Sonnet עם effort גבוה.
- ארכיטקטורה, אבטחה או חקירת שורש: Opus לזמן מוגבל.
- אל תשאיר Opus/high effort כברירת מחדל.

## במהלך העבודה
- חיפוש לפני פתיחת קבצים רבים.
- שינוי מינימלי בלבד.
- לא להמציא נתונים או תוכן.
- שינוי UI: preview וצילום/בדיקה.
- שינוי integration: לבדוק גם environment/live, לא רק build.
- משימה רחבה מדי: לעצור ולפצל.

## Git
- פרויקט משמעותי: branch + PR.
- לא push ישיר ל-main.
- לא merge לפני checks ו-preview.
- PR חייב לומר מה השתנה, מה לא השתנה, מה נבדק ומה עדיין לא אומת.

## טוקנים ו-context
- מעבר לנושא אחר: `/rename` ואז `/clear`.
- המשך אותה משימה עם context ארוך: `/compact Focus on verified decisions, changed files, open risks, and next steps`.
- בדיקה: `/usage`.
- אחרי שני ניסיונות תיקון שלא הצליחו: session חדש עם סיכום קצר.

## Subagents
השתמש רק כאשר מחקר צדדי יוצר הרבה פלט. לא למשימה קטנה ולא כשצריך שיחה רציפה.

## בסיום בקש דיווח כזה
```text
דווח:
1. מה בוצע.
2. מה נבדק מקומית.
3. מה אומת בסביבה חיה.
4. מה עדיין חסום או לא אומת.
5. branch, commit ו-PR.
```

## שלושה כללי ברזל
1. PASS אינו בהכרח READY.
2. unknown עדיף על ניחוש.
3. פחות context וכלים — כל עוד האמינות נשמרת.
