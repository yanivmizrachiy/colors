## חולשות
### כפילות בטעינה הקבועה
אותם עקרונות חוזרים ב-`CLAUDE.md`, `INDEX`, `USER_RULES`, `CHANGE_PROTOCOL` ו-`REQUIREMENTS`:
- לתקן רק מה שהתבקש.
- לא לבצע refactor רחב.
- preview לפני commit.
- tsc/build לפי סוג שינוי.
- GitHub כמקור אמת.
- commit+push בסיום.

לכן המבנה מדורג, אך סדר הקריאה הקבוע עדיין טוען כמה קבצים ארוכים לפני כמעט כל משימה.

### REQUIREMENTS מערבב שכבות
הקובץ מכיל יחד:
- נוהל עבודה.
- מוסכמות מתמטיות ועיצוביות.
- ביצועים.
- אתר `zaviyot` העצמאי.
- פריסה ו-Vercel.
- פרטים היסטוריים מאוד ספציפיים.

הוא מקור אמת טוב לבני אדם, אך אינו מתאים לטעינה מלאה לפני כל שינוי.

### CURRENT_STATE כבד מדי
`CURRENT_STATE.md` מכיל גם state נוכחי וגם מפרט ארכיטקטורה ומפות תוכן. מצב משתנה צריך להיות קצר; מפרטים יציבים צריכים לעבור למסמכי domain.

### TASKS שומר היסטוריה
החלק “בוצע” מצטבר ואינו נדרש לתחילת session. היסטוריה שייכת ל-changelog או commits, לא לזיכרון פעיל.

### PROMPTS הפך מיותר
כאשר `CLAUDE.md` ו-INDEX נטענים אוטומטית, פרומטים שמבקשים שוב לקרוא אותם משכפלים הקשר ומגדילים סיכון להוראות סותרות.

## ממצא מרכזי
המערכת אינה “יותר מדי קבצים” בלבד; הבעיה היא שסדר הקריאה מחייב טעינה של שכבות שחלקן חופפות. הארכיטקטורה הנכונה צריכה לשמור את ההפרדה אך להפוך את רוב הקריאה לתלוית משימה.
