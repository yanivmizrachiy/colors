# חבילת מוכנות לפיילוט `misparim`

עודכן: 2026-07-14

## סטטוס
מסמך זה מכין את הפיילוט בלבד. הוא **אינו מאשר שינוי ב־`misparim`** ואינו משנה אותו. כל יישום בריפו אחר דורש הוראה מפורשת ונפרדת של יניב.

## מטרת הפיילוט
לבדוק באופן מדיד האם ארכיטקטורת זיכרון רזה ומדורגת משפרת את העבודה עם Claude Code בלי להוריד איכות, אמינות או המשכיות.

## ההשערה
הפחתת מידע שנטען בכל session, הפרדת state זמני והעברת שני תחומים ל־path-scoped rules צפויות:
- לצמצם context ושימוש במכסה.
- לקצר את זמן ההבנה הראשוני.
- להפחית חזרות בין קובצי זיכרון.
- לשמור או לשפר את שיעור ההצלחה בפעם הראשונה.

## גבולות קשיחים
- אין שינוי קוד מוצר בפיילוט הזיכרון הראשון.
- אין Skill, Hook, Agent או MCP חדש בשלב הראשון.
- אין קריאה של transcripts, credentials או `.env`.
- אין push ישיר ל־`main`.
- אין merge לפני השוואת המדדים.
- כל שינוי חייב להיות הפיך ב־commit אחד או PR אחד.

## מה יימדד
### יעילות
- זמן מתחילת session עד תחילת עבודה ממוקדת.
- זמן כולל עד תוצאה מאומתת.
- מספר קבצים שנקראו לפני שינוי.
- `/usage` לפני ואחרי המשימה, ככל שהפקודה זמינה בתוכנית המשתמש.
- מספר compact/clear/resume שנדרשו.

### איכות
- מספר סבבי תיקון.
- מספר דרישות שהוחמצו.
- מספר שינויים צדדיים או לא־נדרשים.
- static/build/preview שעברו.
- רגרסיות או תיקוני המשך לאחר PR.

### מעורבות יניב
- מספר שאלות הבהרה שהיה צורך לענות עליהן.
- מספר אישורים ידניים.
- זמן שבו יניב נדרש לעקוב אחרי התהליך.

## עיצוב הניסוי
### שלב A — Baseline ללא שינוי
לבצע שלוש משימות רגילות במבנה הקיים:
1. שינוי תוכן/עיצוב קטן.
2. תיקון קוד בינוני בכמה קבצים.
3. בדיקת repository או bug investigation ללא שינוי רחב.

לכל משימה ממלאים `templates/BASELINE_SESSION_RECORD.md`.

### שלב B — שינוי זיכרון מינימלי
רק לאחר אישור מפורש:
- לקצר `CURRENT_STATE.md` למצב חי בלבד.
- להסיר כפילות בין `CLAUDE.md`, `INDEX`, `USER_RULES` ו־`CHANGE_PROTOCOL`.
- להשאיר שער כניסה קצר.
- לבחור שני תחומים בלבד ל־path rules, למשל:
  - `zaviyot/**`.
  - UI/preview או מתמטיקה/RTL.
- לא לשנות את דרישות המוצר עצמן.

### שלב C — שלוש משימות השוואה
לבצע שלוש משימות דומות בגודל ובסיכון ל־Baseline, באותו מודל וב־effort דומה.

## תנאי הצלחה
הפיילוט ייחשב מוצלח רק אם מתקיימים יחד:
- ירידה של 20% לפחות בזמן ההבנה/פתיחה או בשימוש הנמדד.
- אין עלייה במספר סבבי התיקון.
- אין דרישה קריטית שהוחמצה.
- אין רגרסיה חדשה.
- יניב מדווח שהשיחה ברורה וקלה יותר לניהול.

## תנאי עצירה
עוצרים ומחזירים לאחור אם:
- Claude מפספס חוזה מוצר בגלל מידע שלא נטען.
- נוצר מקור אמת נוסף או כפילות חדשה.
- יש עלייה ברורה בתיקונים או בשאלות הבהרה.
- path rules אינם נטענים באופן צפוי.
- אין שיפור מדיד אחרי שלוש משימות.

## Rollback
- כל שינוי זיכרון יבוצע בענף אחד וב־PR אחד.
- לפני השינוי נשמר snapshot של מבנה הקבצים וה־SHA.
- rollback הוא revert של commit הפיילוט בלבד.
- אין לערב שינויי מוצר באותו PR.

## שלושת סוגי המשימות המומלצים
### P1 — משימה קטנה
שינוי טקסט/עיצוב נקודתי עם preview. מטרתה למדוד overhead של פתיחת session.

### P2 — משימה בינונית
שינוי בכמה קבצים עם typecheck/build/preview. מטרתה למדוד הבנת תלויות ושמירת scope.

### P3 — חקירת bug
חקירה read-only תחילה, ניסוח root cause ורק לאחר מכן תיקון מינימלי. מטרתה למדוד איכות reasoning ומספר סבבי התיקון.

## תוצרי הפיילוט
- שלושה רישומי baseline.
- שלושה רישומי post-change.
- טבלת השוואה אחת.
- החלטה: הרחבה, תיקון או rollback.
- רק אם הבעיה עדיין חוזרת שלוש פעמים: בחינת Skill או Hook אחד.

## שער האישור
המשפט שמאפשר להתחיל שינוי ב־`misparim` צריך להיות מפורש, למשל:

> מאשר לבצע את שלב B של פיילוט הזיכרון ב־misparim לפי `docs/PILOT_MISPARIM_READINESS.md`, בענף ו־PR בלבד, בלי שינוי קוד מוצר ובלי merge אוטומטי.
