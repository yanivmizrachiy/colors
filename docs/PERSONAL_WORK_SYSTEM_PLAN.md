# תוכנית מערכת העבודה האישית — מסגרת מחקרית

> מסמך זה הוא שלד לתוכנית הסופית. הוא אינו אישור ליישום.

## 1. תמונת המצב
- פרויקטים פעילים.
- סוגי משימות עיקריים.
- כלים ואינטגרציות.
- דפוסי הצלחה.
- חיכוכים וטעויות חוזרות.
- baseline של זמן, תיקונים, בדיקות והקשר.

## 2. עקרונות התכנון
- מינימום מידע שנטען תמיד.
- מקור אמת יחיד לכל סוג מידע.
- אוטומציה רק לצורך מוכח.
- פעולה מסוכנת דורשת guard ו-rollback.
- כל משימה מסתיימת בהוכחת הצלחה.
- הבחנה בין משימה קטנה, בינונית וגדולה.
- התאמה נפרדת לכל פרויקט לצד ליבה משותפת.

## 3. ארכיטקטורת זיכרון מוצעת
ייבנו חלופות להשוואה:
- זיכרון גלובלי מינימלי.
- שער פרויקט קצר.
- מצב נוכחי נפרד.
- Rules לפי תחום או path.
- Skills לתהליכים חוזרים.
- מחקר והיסטוריה מחוץ להקשר הקבוע.

לכל חלופה ייבדקו: תועלת, עלות context, תחזוקה, סיכון ויכולת rollback.

## 4. מחזור עבודה מוצע
### פתיחת משימה
- זיהוי ריפו, branch ומצב Git.
- סיווג גודל וסיכון.
- קריאת מקורות אמת רלוונטיים בלבד.

### חקירה ותכנון
- איתור references וקוד קיים.
- זיהוי תלויות, סיכונים וקריטריוני קבלה.
- תכנון בדיקות ו-preview לפני שינוי.

### ביצוע
- שינוי ממוקד.
- commit ברור.
- שמירת גבולות ריפו ו-secrets.

### אימות
- typecheck/lint/tests/build.
- בדיקות אינטגרציה וסביבת פריסה לפי הצורך.
- preview ובדיקה חזותית בפרויקטי UI.
- בדיקת דרישות המשתמש אחת-אחת.

### סיום והמשכיות
- סיכום מה השתנה ומה נבדק.
- רישום סיכונים פתוחים.
- handoff קצר למשימה הבאה.
- clear/compact לפי מצב ההקשר.

## 5. רכיבים עתידיים לבדיקה
- Memory גלובלי.
- תבנית `CLAUDE.md` לפרויקט.
- Rules לפי path.
- Skills.
- Hooks/guards.
- Subagents.
- MCP/Plugins.
- scripts ו-tests.
- Git/PR/deploy workflow.

כל רכיב יקבל: בעיה, ראיות, חלופות, סיכון, עלות תחזוקה ומדד הצלחה.

## 6. מדיניות מודלים והקשר
ייבדקו:
- איזה מודל מתאים לתכנון, קוד, מחקר וביקורת.
- מתי להשתמש ב-effort נמוך או גבוה.
- מתי לפתוח session חדש.
- מתי להשתמש ב-compact, clear, handoff או subagent.
- כיצד לצמצם MCP וכלים שאינם בשימוש.

## 7. מדדי הצלחה
- זמן עד פתרון עובד.
- מספר סבבי תיקון.
- שיעור דרישות שהושלמו בפעם הראשונה.
- תקלות לאחר merge ופריסה.
- זמן התערבות ידני של יניב.
- גודל context וצריכת tokens.
- שיעור בדיקות ו-preview שהושלמו.
- מספר רכיבים אוטומטיים שנשארו שימושיים לאחר חודש.

## 8. תוכנית פיילוט
תיבחר רק לאחר השלמת המחקר:
- פרויקט אחד.
- baseline לפני שינוי.
- שינוי מינימלי והפיך.
- תקופת בדיקה.
- השוואת מדדים.
- החלטת הרחבה, תיקון או rollback.

## 9. תנאי מעבר ליישום
1. המחקר הושלם בכמה פרויקטים.
2. נאסף מידע מקומי מסונן.
3. לכל המלצה יש ראיות ומדד הצלחה.
4. יניב אישר את סדר העדיפויות.
5. הוגדר rollback.
6. היישום ייעשה בפרויקט פיילוט בלבד.
