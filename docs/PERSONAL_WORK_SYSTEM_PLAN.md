# תוכנית העבודה האישית של יניב עם Claude Code — קלוטקורד גרסה 2

עודכן: 2026-07-15

> זהו מסמך הביצוע המרכזי של קלוטקורד. הוא מרכז את כל דרישות יניב, מסקנות המחקר ותוכנית היישום. הוא אינו דף כללים מחייב; מקור הכללים היחיד הוא `CLAUDE.md`.

## 1. מטרת התוכנית
לאפשר ליניב לתת הוראות קצרות ולקבל עבודה אמיתית, מדויקת, בדוקה ובטוחה—תוך שמירת הדרישות שלו לאורך זמן, צמצום הסברים חוזרים, חיסכון ב־context ובטוקנים ושימוש מדורג ב־Skills, Hooks, Plugins, Subagents, Git, CI ו־preview.

היעד אינו לייצר הרבה עבודה או מקסימום אוטומציה. היעד הוא לייצר את העבודה הנכונה:
- מינימום מאמץ ידני מצד יניב.
- מינימום מידע וכלים שאינם נחוצים למשימה.
- מקסימום שמירה על דרישות וחוזי מוצר.
- תוצאה אמיתית ולא מצג הצלחה.
- יכולת להבין, לבדוק ולהחזיר לאחור כל שינוי.

## 2. בסיס המחקר
התוכנית מבוססת על פרויקטי הליבה הגדולים של יניב ועל דפוסים שחזרו בעבודה אמיתית:
- מערכות Web גדולות ב־TypeScript/Next.js.
- מערכות חינוך עם RTL, מתמטיקה, A4, PDF ומצגות.
- מערכות רגישות עם תלמידים, DB, הרשאות ואינטגרציות.
- מערכות שליטה מקומיות עם מכשירים, PowerShell ו־runtime אמיתי.
- קליטת PDF/DOCX וקבצים אמיתיים.
- עבודה עם GitHub, Vercel, Actions, Lovable, AI Studio וכלי AI נוספים.

המסקנה אינה שכל הפרויקטים צריכים אותה תבנית. המסקנה היא שיש ליבה משותפת אחת, ומעליה התאמה לפי גודל וסיכון.

## 3. חוזה ההתנהגות שאני דורש מ־GPT ומ־Claude

### 3.1 תלמד אותי ואל תתחיל בכל פעם מחדש
כאשר יניב נותן דרישה שעשויה להיות עמידה, Claude צריך לזהות אותה כמועמדת לזיכרון ולא להתייחס אליה כמשפט זמני בלבד.

לפני שמירה:
1. לחפש אם הדרישה כבר קיימת.
2. לבדוק אם היא סותרת דרישה קודמת.
3. להציע ניסוח ברור וקצר.
4. להסביר מה היא מחליפה.
5. להסביר היכן היא תישמר.
6. לשמור רק לאחר אישור יניב.

אין לצבור כמה גרסאות פעילות של אותה דרישה. ההוראה המאושרת האחרונה מחליפה את הקודמת.

### 3.2 אל תשמור הכול באותו סוג זיכרון
יש להבחין בין:
- העדפה אישית.
- כלל עבודה קבוע.
- חוזה מוצר.
- החלטה ארכיטקטונית.
- מצב נוכחי.
- משימה פתוחה.
- היסטוריה.
- ראיה או תוצאת בדיקה.
- מידע רגיש שאסור לשמור.

State זמני אינו נכנס לדף הכללים. רשימת משימות אינה זיכרון קבוע. דוח AI או אחוז מוכנות אינם אמת נוכחית בלי אימות מול Git, בדיקות ו־runtime.

### 3.3 אל תמציא שום דבר
אין:
- נתוני דמו שמוצגים כאמיתיים.
- כפתורים שלא עובדים.
- הצלחה שלא נבדקה.
- תוכן לימודי מומצא.
- מספרים או מקורות שהומצאו.
- טענה שמוצר מוכן רק מפני שה־build עבר.
- טענה שקובץ, branch, PR או deployment קיימים בלי בדיקה.

כאשר אין הוכחה, משתמשים בניסוח מדויק: לא אומת, לא ידוע, חסר מקור, נבדק מקומית בלבד או עדיין לא נבדק חי.

### 3.4 אל תסכים איתי אוטומטית
Claude צריך להתנהג כמומחה בכיר:
- לבדוק אם הרעיון נכון.
- לחפש פתרון פשוט יותר.
- להציג סיכון ועלות בטוקנים, זמן ותחזוקה.
- להזהיר ממערכת מורכבת מדי.
- לעצור כיוון לא יעיל.
- לא להמליץ על כלי רק משום שהוא חדש או מרשים בסרטון.

המטרה היא ביקורת מקצועית, לא הסכמה.

### 3.5 אל תשאל שאלות שאפשר לפתור לבד
לפני שאלה ליניב:
1. לבדוק את הריפו.
2. לחפש בקבצים.
3. לבדוק commits ו־PRs.
4. לקרוא את ה־state העדכני.
5. לבדוק אם יניב כבר נתן את התשובה.
6. לבדוק תיעוד רשמי כאשר מדובר בכלי עדכני.

שואלים רק על החלטה שרק יניב יכול לקבל. אין לשאול שוב שאלה שכבר נענתה.

### 3.6 היה עצמאי, אך לא חסר גבולות

בפעולה מסוכנת או בלתי הפיכה—מכינים הכול ועוצרים בשער ההחלטה.

עצמאות אינה הרשאה:

- להתקין כלי חיצוני.

3. מה אומת חי.

אין לשלוח יומן פנימי ארוך כאשר סיכום קצר מספיק.

## 5. ארכיטקטורת הזיכרון

### 5.1 זיכרון גלובלי של יניב
יכיל רק העדפות שחוצות פרויקטים:
- תקשורת בעברית.
- עבודה אמיתית בלבד.
- אין המצאת תוכן, נתונים או הצלחה.
- GitHub הוא מקור האמת לקוד.
- שינוי ממוקד בלבד.
- דרישה אחרונה מאושרת מחליפה דרישה קודמת.
- אין secrets או נתוני תלמידים בזיכרון ציבורי.
- בדיקה חזותית נדרשת למתמטיקה, RTL, A4, PDF ו־UI.

יעד: 20–40 שורות בלבד. הזיכרון האישי המלא צריך להיות פרטי.

### 5.2 `CLAUDE.md` בפרויקט
יעד: 80–150 שורות, ובכל מקרה פחות מ־200 ככל האפשר.

יכיל רק:
1. זהות המוצר והריפו הקנוני.
2. גבולות שינוי.
3. מקורות אמת.
4. פקודות build/test/preview.
5. חוזי בטיחות קריטיים.
6. הגדרת done.
7. הפניות למסמכים או Rules לפי צורך.

לא יכיל:
- changelog.
- רשימת “בוצע”.
- state זמני.
- snapshots של חשבון, חומרה או אחוזי מוכנות.
- מפת מוצר מלאה.
- tutorial ארוך.
- אותה דרישה בכמה ניסוחים.

### 5.3 `CURRENT_STATE.md`
יכיל רק:

- מה עובד ומה חסום.

- מה אומת חי ומה לא.

- תאריך ומקור ראיה.

יעד: 30–60 שורות.

### 5.4 דרישות מוצר
מקור קנוני אחד לכל concern או data/schema קנוני:
- תוכן לימודי.
- חוזי RTL ומתמטיקה.
- DB/schema.
- הרשאות ו־roles.
- integration contracts.
- A4/PDF/print.

מידע דטרמיניסטי עובר ל־test, verifier, schema או guard במקום להישאר רק בטקסט.

### 5.5 Rules לפי path
רק כאשר קיימים domains אמיתיים:
- UI/CSS.
- math/RTL.
- print/A4/PDF.
- DB/auth/security.
- Moodle/LTI.
- subproject עצמאי.

Rule אינו משכפל workflow גלובלי ואינו מחזיק state.

### 5.6 Auto Memory
מותר:
- פקודת debug שהתגלתה.
- gotcha טכנית שקשה לגלות.
- preference חוזרת שאושרה.
- נתיב או כלי מקומי שאינו רגיש.

אסור:

- secrets.

- logs גדולים.
- מידע תלמידים.
- החלטה שסותרת Git.

ביקורת `/memory` אחת לשבועיים בפרויקט פעיל.

## 6. כיצד Claude לומד דרישה חדשה
1. מזהה שהמשפט עשוי להיות דרישה עמידה.
2. מסווג: העדפה גלובלית, כלל פרויקט, חוזה מוצר, החלטה, state, משימה או מידע רגיש.
3. מחפש כפילות, ניסוח דומה, סתירה או test קיים.
4. מציג ליניב ניסוח קנוני, scope, מיקום ומה יוחלף.
5. כותב רק לאחר אישור.
6. מעדכן את הדרישה הקיימת במקום לצבור גרסאות.
7. מקשר ל־test, verifier, schema, permission, Rule, Skill או Hook כאשר צריך.

לדרישה חשובה ניתן לשמור:
- `id` יציב.
- סוג ו־scope.
- ניסוח קנוני.
- מקור ותאריך.
- סטטוס: proposed / approved / verified / superseded / revoked.
- רמת רגישות.
- references לקוד ולבדיקות.
- תאריך review כאשר היא עלולה להתיישן.

## 7. ארבעה מסלולי עבודה

### Fast Lane — שינוי קטן
מתאים לניסוח, CSS נקודתי, label או bug מקומי וברור.

תהליך:
1. אימות repo ו־scope.
2. חיפוש ממוקד.
3. שינוי מינימלי.
4. בדיקה ממוקדת.
5. קריאת diff.
6. PR קטן או commit לפי מדיניות הפרויקט.

מודל: Sonnet, effort רגיל.

### Standard Lane — שינוי בינוני
מתאים לרכיב, flow, כמה קבצים או integration מוגבל.

תהליך: Explore → תוכנית קצרה → implementation → tests + preview → PR עם root cause, scope ו־rollback.

מודל: Sonnet, effort גבוה.

### High-Risk Lane — שינוי מסוכן
מתאים ל־DB, auth, permissions, תלמידים, Moodle/LTI, shell, מכשיר, production או שינוי רוחבי.

תהליך:
1. evidence.
2. risk analysis.
3. plan ו־rollback.
4. branch/worktree מבודד.
5. draft PR.
6. review עצמאי.
7. environment/live validation.
8. merge מפורש בלבד.

מודל: Opus לתכנון, root cause וביקורת; Sonnet לביצוע שלבים ברורים.

### Research Lane — חקירה בלבד
מתאים ל־repo audit, CI/logs, OCR/render/artifact או בדיקת PR.

ברירת מחדל: read-only. אין PR מוצר רק כדי להפיק artifact כאשר קיימת חלופה מקומית.

## 9. חיסכון ב־context ובטוקנים

### Sessions
- משימה משמעותית אחת בכל session.
- `/rename` לפני מעבר.
- `/clear` בין משימות לא קשורות.
- `/compact Focus on verified decisions, changed files, open risks, and next steps` כאשר אותה משימה נמשכת.
- לאחר שני ניסיונות תיקון כושלים: לעצור, לסכם ולפתוח session נקי.
- `/btw` לשאלה קצרה שאינה צריכה להיכנס להיסטוריה.

### קריאת מידע
- Grep/search לפני פתיחת קובץ גדול.
- טווח שורות רלוונטי במקום קובץ מלא.
- אין קריאת כל הריפו או כל requirements כברירת מחדל.
- logs ו־CI ארוכים עוברים ל־Subagent או artifact מסונן.

### מודלים
- Sonnet כברירת מחדל.
- effort רגיל ל־Fast Lane.
- effort גבוה ל־Standard/debugging.
- Opus רק לארכיטקטורה, root cause, אבטחה, migration או review מסוכן.
- Agent קל למשימות מיון או סיכום ללא החלטה מורכבת.

### מדידה
בכל פיילוט מודדים:
- `/usage` ו־`/context`.
- זמן עד root cause.
- זמן עד תוצאה מאומתת.
- מספר הקבצים שנקראו.
- מספר ניסיונות התיקון.
- רגרסיות.

אין לטעון לחיסכון בלי baseline והשוואה.

## 10. Git, PR ו־Merge Gates

### לפני שינוי
בודקים:
- repository.
- remote.
- branch.
- working tree.
- מצב `main`.
- PR או עבודה מקבילה.

### שינוי קטן
PR קטן, commit אחד או שניים, diff review ובדיקות ממוקדות.

### שינוי בינוני
PR עם:
- הבעיה ושורש התקלה.
- מה השתנה.
- מה לא השתנה.
- tests/preview.
- rollback.

### שינוי מסוכן
- draft PR.
- reviewer או agent עצמאי.
- זמן קירור לפני merge.
- evidence חי.
- merge מפורש.

### איסורים
- commits ניסיוניים ישירות ל־`main`.
- force push.
- `reset --hard`.
- `git clean`.
- PR של נושאים רבים לא קשורים.
- PR מחקרי ללא תאריך תפוגה.
- merge אוטומטי רק משום שה־build ירוק.

### סטטוסים נפרדים
- implemented.
- tests passed.
- preview/environment verified.
- live verified.
- release ready.

## 12. טיפול בתיקון כושל
לאחר ניסיון תיקון כושל אחד:
- לעצור.

- לחפש source of truth, override, wrapper, cache או שכבה כפולה.

לאחר שני ניסיונות כושלים:
- לסכם ממצאים.
- לנקות context.
- לפתוח session חדש.
- להתחיל מחקירת שורש.

## 13. Skills — סדר הבנייה
לא מתקינים Skill pack. בונים או מתקינים Skill אחד בכל פעם.

### S1 — `capture-requirement`
ה־Skill הראשון והחשוב ביותר.

תהליך:
1. סיווג הדרישה.
2. חיפוש כפילות וסתירה.
3. הצעת ניסוח קנוני ומיקום.
4. הצגת מה יוחלף.
5. כתיבה רק לאחר אישור.
6. הצעת test/guard/Rule/Skill כאשר טקסט אינו מספיק.

בטיחות:
- `disable-model-invocation: true`.
- הפעלה ידנית בלבד.
- אין שינוי קוד מוצר.
- אין מידע רגיש.
- אין יצירת דף כללים נוסף.

### S2 — `safe-change`
Preflight, scope, tests, diff ו־PR report.

מותר: Read/Grep/Glob, Git read/status/diff ופקודות בדיקה מאושרות.

אסור: merge, deploy, DB write ו־force operations.

### S3 — `ui-verify`

### S5 — `handoff`
סיכום מסונן בין sessions ומחשבים: מטרה, מצב, files changed, tests, branch/PR, החלטות, חסימות והצעד הבא. ללא transcript גולמי.

### S6 — `ingest-real-files`

### S7 — `math-rtl-verify`
רק לאחר צורך מוכח בכמה פרויקטים: סימן לפני מספר, LTR בתוך RTL, גרפים/SVG, מקור תוכן ו־A4/print.

## 14. תנאי יצירת Skill או התקנת Plugin
רכיב נוצר או מותקן רק כאשר מתקיים לפחות אחד:
- workflow חזר שלוש פעמים.
- עלות הטעות גבוהה במיוחד.
- התהליך רב־שלבי וקשה לזכור.
- הוא חוסך context קבוע משמעותי.

לפני יצירה או התקנה מתעדים:
1. בעיה אמיתית.
2. תדירות וחומרה.
3. חלופה פשוטה יותר.
4. מקור ו־commit/tag מדויק.
5. קוד ותלויות.
6. כלים והרשאות.
7. auto-invocation או הפעלה ידנית.
8. עלות context וטוקנים.
9. מידע שעוזב את המחשב.
10. מדד הצלחה.
11. דרך הסרה ו־rollback.

אין להתקין הרחבה רק משום שהופיעה בסרטון.

## 15. Permissions, Hooks ו־Sandbox

### Deny
- `.env`, credentials, tokens, cookies ו־private keys.
- force push, `reset --hard`, `git clean` ומחיקה רחבה.
- כתיבה מחוץ ל־root.

### Allow
- `git status`, `git diff`, `git log`.
- tests/typecheck/build ידועים.
- קריאה לא־רגישה.

### Ask
- push.
- merge.
- deploy.
- DB.
- auth והרשאות.
- פעולות system/device.

יש להסיר הרשאות project-local רחבות למיזוג כאשר אינן מוצדקות.

Hook נבנה רק אם Permission או test אינם מספיקים. מועמדים: חסימת פעולות הרסניות, secrets ו־push ישיר ל־`main`. אין Hook שמריץ build כבד אחרי כל edit.

Sandbox נבחן רק לאחר inventory מקומי; ב־Windows יש לבדוק WSL2 כאשר נדרש גבול OS אמיתי.

## 16. Subagents ו־Agent Teams
Subagent מתאים ל־audit read-only, CI/logs, security review וחיפוש רחב. ברירת מחדל: סוכן אחד, תוצר קצר וללא edits.

Agent Teams אינם ברירת מחדל. הם מותרים רק כאשר:
- המשימות עצמאיות באמת.
- אין עריכה באותו קובץ.
- לכל teammate תוצר ברור.
- עלות ה־context מוצדקת.

## 17. אוטומציות מתוזמנות
Scheduled task, Action או Agent אינם רשאים:
- לשכתב `CLAUDE.md` או `RULES.md`.

אוטומציה מתוזמנת יכולה לבדוק ולהציע. היא אינה מחליטה במקום יניב.
