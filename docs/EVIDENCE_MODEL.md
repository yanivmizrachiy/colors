# מודל ראיות ורמות ודאות

## למה צריך מודל ראיות
המיזם עוסק בשיפור מערכת עבודה מורכבת. כדי להימנע מהמלצות כלליות או מסקנות שגויות, כל ממצא נשמר עם מקור, רמת ודאות והשלכה מוגדרת.

## סוגי קביעה
### עובדה
מידע שנצפה ישירות בקובץ, commit, PR, הגדרה, פלט פקודה או תיעוד רשמי.

### הסקה
מסקנה הגיונית שמבוססת על כמה עובדות. יש לציין במפורש שזו הסקה.

### השערה
אפשרות שעדיין דורשת אימות נוסף, למשל באמצעות נתונים מקומיים או שיחה עם יניב.

### החלטה
בחירה שאושרה לאחר השוואת חלופות. החלטה נשמרת ב-`DECISIONS.md`, לא בתוך דוח מחקר בלבד.

## רמות ודאות
- **A — מאומת ישירות:** מקור ראשוני ברור ועדכני.
- **B — נתמך חזק:** כמה מקורות תואמים או דפוס שחוזר בכמה מקרים.
- **C — סביר:** מקור חלקי או הסקה הגיונית שדורשת אימות.
- **D — פתוח:** השערה, מידע חסר או סתירה בין מקורות.

## משקל מקור
1. תיעוד רשמי או קוד פעיל.
2. בדיקה אוטומטית, CI או פלט runtime.
3. commit/PR עם הסבר ואימות.
4. README או מסמך פרויקט עדכני.
5. הערת שיחה או זיכרון לא מאומת.

## פורמט ממצא
כל ממצא צריך לכלול:

```text
ID:
תאריך:
פרויקט:
סוג: עובדה / הסקה / השערה
רמת ודאות: A / B / C / D
מקור:
מה נצפה:
למה זה חשוב:
השפעה על איכות:
השפעה על זמן:
השפעה על הקשר/טוקנים:
סיכון:
מידע חסר:
מועמד עתידי: Memory / Rule / Skill / Hook / Agent / Script / Test / Process / None
סטטוס: פתוח / מאומת / נדחה / הועבר להחלטה
```

## כללי מניעת הטיה
- PR מוצלח אינו מוכיח שהתהליך היה יעיל; צריך לבדוק גם תיקוני המשך.
- כמות טסטים אינה מוכיחה כיסוי של פרודקשן.
- קובץ `CLAUDE.md` קצר אינו בהכרח טוב אם חסר בו מידע קריטי.
- קובץ ארוך אינו בהכרח רע אם נטען רק לפי צורך.
- דפוס ייחשב חוצה-פרויקטים רק לאחר שנצפה בשני פרויקטים לפחות.
- המלצה בעלת סיכון גבוה דורשת ראיות חזקות יותר.

## מעבר מממצא להמלצה
ממצא יכול להפוך להמלצה רק לאחר שנבדקו:
1. תדירות הבעיה.
2. חומרת הנזק.
3. עלות הזמן וההקשר.
4. חלופות פשוטות יותר.
5. סיכון תחזוקה.
6. דרך למדוד הצלחה.
7. אפשרות rollback.
