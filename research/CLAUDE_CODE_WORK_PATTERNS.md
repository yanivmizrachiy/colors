# דפוסי העבודה של יניב עם Claude Code

עודכן: 2026-07-15

> מסמך מחקר מסונן בלבד. הוא אינו דף כללים ואינו כולל secrets, מידע אישי או פרטים מתוך ריפוים פרטיים. מקור הכללים היחיד בקלוטקורד הוא `CLAUDE.md`.

## מטרת המחקר
להבין לא רק מה הפרויקטים עושים, אלא איך יניב עובד עם Claude Code בפועל: כיצד דרישה נולדת, איך היא הופכת לקוד, מה נבדק, היכן נוצרים תיקוני המשך, ואילו שלבים חוזרים בין מוצרים.

## דפוסי עבודה חזקים שכדאי לשמר

### 1. אמת לפני דמו
במרבית הפרויקטים חוזר חוזה ברור:
- אין נתונים, כפתורים או סטטוסים מזויפים.
- מידע חסר מוצג כחסר או לא מאומת.
- build שעבר אינו מוצג כהוכחה למערכת חיה.
- פרויקט רגיש משתמש לעיתים בשערי release נפרדים.

### 2. תרגום הערת משתמש לחוזה קבוע
יניב נותן הערות מדויקות מאוד על:
- ניסוח עברי ומתמטי.
- RTL ו־A4.
- גרפים, SVG, תוויות וחפיפות.
- כפתורי העתקה, PNG ו־PDF אמיתיים.
- mobile, PWA וקישורים ציבוריים.
- פשטות הממשק והסרת כפתורים מיותרים.

בפרויקטים הבשלים ההערה הופכת ל־test, verifier או כלל קנוני.

### 3. אימות רב־שכבתי
נמצאו שילובים של:
- typecheck, lint, unit tests ו־build.
- browser verification.
- בדיקת מאות דפי A4 בכמה viewports.
- stress runs ו־seeded generation.
- בדיקות production/browser/PWA/offline.
- live verification נפרד כאשר נדרשים DB, Moodle, hosting או מכשיר אמיתי.

### 4. תיאורי commit ו־PR עשירים
בפרויקטים מרכזיים תיאור השינוי כולל:
- root cause.
- מה השתנה ומה לא השתנה.
- בדיקות שעברו.
- גבולות סיכון.
- מצב live או blockers.

זהו נכס חשוב ל־handoff, audit ו־rollback.

### 5. חיפוש פתרון שורש
כאשר התקלה מובנת היטב, Claude מיישם לעיתים פתרונות מערכתיים חזקים:
- source of truth יחיד.
- cache versioning מרכזי במקום תיקון ידני לכל דף.
- forwarding של options דרך שרשרת wrappers.
- storage provider מותאם ל־serverless.
- guard שמונע חזרת ניסוח, קובץ legacy או מצב שגוי.

## דפוסי חיכוך וסיכון

### 1. קצב מיזוג מהיר מדי
בפרויקטים מסוימים PR נוצר ומוזג בתוך דקה או דקות בודדות, ללא reviewer נפרד. הדבר יעיל לשינוי קטן, אך מסוכן כאשר יש:
- DB/migration.
- הרשאות ואימות.
- שינוי רוחבי במאות קבצים.
- הסרת ארכיטקטורה.
- release או deployment.

### 2. PRs גדולים מאוד
נמצאו PRs עם עשרות commits ושינויים מערכתיים רבים באותה יחידה. הם כוללים לעיתים UI, PWA, CI, state, rules, performance וניקוי יחד.

המחיר:
- קשה להבין מה גרם לרגרסיה.
- קשה לבצע review אמיתי.
- rollback מחזיר גם שינויים טובים שאינם קשורים.
- context של Claude גדל מאוד.

### 3. רצף commits ישירות ל־main
בפרויקטי תשתית מסוימים לא נעשה שימוש ב־PR, ונוצרו commits רבים בהפרש של שניות, כולל פעולות חוזרות על אותם קבצים.

הסיכון:
- אין checkpoint reviewable לפני פרסום.
- העבודה האינטראקטיבית הופכת להיסטוריית main.
- קשה להבחין בין ניסוי, כשל ותוצאה סופית.

### 4. תיקון על תיקון
נמצאו מקרים שבהם:
- אותה תקלה תוקנה בכמה commits רצופים.
- override נוסף בסוף קובץ במקום איחוד הלוגיקה.
- branch או PR ראשון הוחלף בענף נקי שני.
- פתרון זמני יצר workflow או artifact נוסף שנדרש לנקות.

כלל מחקרי מוצע: לאחר ניסיון תיקון כושל אחד, הניסיון הבא מתחיל ב־root-cause review ובקריאת כל המסלול הרלוונטי.

### 5. זיכרון כבד מדי
דף כללים יחיד פותר כפילויות, אך בפרויקטים גדולים הוא גדל עד אלפי מילים וכולל:
- חשבונות ופרטי runtime.
- architecture.
- כל היכולות.
- UI.
- state.
- changelog.
- הוראות הפעלה.

נמצא מקרה שבו נדרש PR ייעודי לקונסולידציה ולתקרת גודל קשיחה.

### 6. state, proof ו־audit בתוך Git
בחלק מהפרויקטים runtime outputs, דוחות proof או status snapshots נכתבו לתוך הריפו. לאחר מכן נדרש ניקוי כדי להעביר state חי מחוץ ל־Git.

הסיכון:
- state מתיישן.
- מידע פרטי על מכשיר או משתמש עלול להישמר.
- כל session קורא ראיה שאינה עוד נכונה.

### 7. שימוש ב־GitHub כמעבדת חקירה
נמצאו PRs זמניים לצילום, OCR/רינדור, forensic audit או הפעלת workflow חד־פעמי. חלקם נוקו היטב, אך נוצר גם PR בטעות וריבוי PRs שלא נועדו למיזוג.

נדרשת תבנית קבועה למחקר זמני:
- prefix מוסכם.
- draft כברירת מחדל.
- label `research-only`.
- ללא merge.
- תאריך תפוגה.
- ניקוי אוטומטי או checklist סגירה.

### 8. preference ל־"אפס צעדים ידניים"
יניב מעדיף שהמערכת תבצע את העבודה בעצמה. הדבר מוביל לפתרונות מועילים כגון setup ו־migration אוטומטיים, אך עלול ליצור סיכון כאשר האוטומציה:
- משנה DB בזמן גישה ראשונה.
- מתקינה autostart או הרשאות מערכת.
- מגדירה secrets או hosting.
- פועלת ללא preview של ההשפעה.

יש להבחין בין אוטומציה reversible ובטוחה לבין פעולה חד־פעמית מסוכנת שדורשת אישור.

### 9. עבודה בכמה סביבות וכלים
העבודה של יניב עוברת בין:
- Claude Code.
- ChatGPT/Codex.
- Lovable ו־AI Studio.
- GitHub ו־GitHub Actions.
- Vercel, Render ו־GitHub Pages.
- Windows/PowerShell.
- Termux/Android.
- browser/PWA/Electron.

הסיכון המרכזי הוא environment drift: התיעוד מתאר סביבה אחת והקוד או הפריסה הפעילה נמצאים באחרת.

### 10. encoding ו־PowerShell
הפעלה של סקריפט עברי דרך PowerShell נכשלה בגלל encoding, והטקסט המשובש הפך לשגיאת parser. זה מלמד שכל כלי Windows עתידי צריך:
- UTF-8 מפורש.
- בדיקת parser לפני הפצה.
- לוג באנגלית טכנית כאשר console encoding אינו ודאי.
- launcher קצר ללא טקסט עברי קריטי בקוד.

## סיווג משימות עתידי

### Fast lane
שינוי קטן, הפיך, ללא נתונים או הרשאות:
- ניסוח.
- CSS נקודתי.
- label.
- תיקון תצוגה ממוקד.

תהליך: branch קצר → בדיקה ממוקדת → PR קטן → merge לאחר diff review.

### Standard lane
שינוי feature או bug שחוצה כמה קבצים:
- explore.
- plan קצר.
- implementation.
- tests + browser.
- PR עם root cause ו־scope.

### High-risk lane
- DB/migrations.
- authentication/permissions.
- תלמידים או מידע רגיש.
- מכשיר, admin, autostart או shell.
- deploy/production.
- שינוי רוחבי גדול.

תהליך: evidence → plan → backup/rollback → isolated branch/worktree → review נפרד → live validation → merge מפורש.

### Research lane
- audit.
- OCR/render/artifact.
- בדיקת קבצים או PR ללא שינוי מוצר.

תהליך: read-only ככל האפשר; artifact מחוץ ל־main; ללא workflow זמני אלא אם אין חלופה.

## מדדים שצריך להתחיל לאסוף
- זמן מהבקשה עד root cause.
- מספר commits ו־PRs למשימה אחת.
- מספר ניסיונות תיקון לפני הצלחה.
- זמן יצירת PR עד merge.
- מספר קבצים ושורות בכל PR.
- מספר tests לעומת live checks.
- כמה פעמים rules/state עודכנו אחרי הקוד ולא יחד איתו.
- כמה PRs נסגרו כ־superseded/research/noop.
- כמה פעולות דרשו אישור משתמש או פעולה ידנית.
- context/usage בתחילת ובסוף session.

## מסקנת ביניים
יניב כבר עובד עם Claude Code ברמה מתקדמת מאוד: הוא משתמש בו כמפתח, חוקר, בודק, מעצב ומפעיל מערכות. הבעיה אינה מחסור בכוח, אלא צורך במערכת שמחליטה מתי להיות מהירה ומתי לעצור, חוקרת לפני עריכה, מנהלת דרישות מתמשכות, מפרידה בין state לכללים, ומפחיתה PR/commit noise בלי לפגוע באוטונומיה.
