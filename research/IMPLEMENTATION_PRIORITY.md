# סדר עדיפויות ליישום עתידי

## עיקרון
אין ליישם לפי “כמה מתקדם זה נשמע”, אלא לפי שילוב של השפעה, ראיות, סיכון ועלות תחזוקה.

## Priority 0 — מדידה לפני שינוי
### P0.1 Baseline
- שלוש משימות מייצגות.
- אותו מודל ו־effort להשוואה.
- רישום זמן, שימוש, תיקונים, בדיקות והתערבות יניב.
- ללא baseline אין טענה אמינה לחיסכון.

### P0.2 Inventory מצומצם
- גרסת Claude Code.
- שמות Skills/Hooks/Agents/Plugins/MCP בלבד.
- שמות קובצי settings וגדלים בלבד.
- אין transcripts, secrets, cookies או ערכי env.

## Priority 1 — זיכרון ו־context
### P1.1 שער פרויקט קצר
השפעה צפויה: גבוהה. סיכון: בינוני. עלות תחזוקה: נמוכה.

### P1.2 `CURRENT_STATE` חי וקצר
השפעה צפויה: גבוהה. סיכון: נמוך. עלות תחזוקה: נמוכה.

### P1.3 שני path rules בלבד בפיילוט
השפעה צפויה: בינונית-גבוהה. סיכון: בינוני. עלות תחזוקה: בינונית.

לא מתחילים ביותר משני domains כדי שניתן יהיה לייחס את התוצאה לשינוי ברור.

## Priority 2 — תהליך עבודה
### P2.1 Task brief קצר ואחיד
- משימה, היקף, אזורים מוגנים, הצלחה ובדיקות.
- לא super-prompt.

### P2.2 PASS מול READY
- static/build אינם הוכחה ל־environment/live.
- כל PR מדווח מה בוצע, מה נבדק ומה עדיין לא אומת.

### P2.3 PR slicing
- פיצול פיצ'רים גדולים לשלבים עצמאיים.
- checkpoint ו־rollback לכל שלב.

## Priority 3 — Guards דטרמיניסטיים
רק לאחר שהפיילוט מוכיח שהבעיה חוזרת:
1. חסימת ריפו שגוי ופקודות הרסניות.
2. חסימת secrets ונתוני תלמידים.
3. בדיקת branch לפני commit/push.
4. test selection לפי קבצים ששונו.

## Priority 4 — Skills
Skill ייבנה רק לאחר שאותו workflow בוצע לפחות שלוש פעמים ונמצא יציב:
1. `safe-change`.
2. `ui-verify`.
3. `repo-audit` read-only.
4. `ingest-real-files` עם dry-run ו־hash.
5. `handoff`.

## Priority 5 — אופטימיזציה של CI
רק בפרויקטים שבהם נמדדה עלות ממשית:
- build-once/test-many.
- path-aware selection עם fallback מלא.
- artifacts זמניים במקום דוחות committed.
- הפרדת quick PR gates מ־deep scheduled audit.

## דברים שאינם בעדיפות כרגע
- Agent teams כברירת מחדל.
- עשרות commands או agents.
- זיכרון גלובלי ארוך.
- Hook שמריץ build מלא אחרי כל עריכה.
- שכתוב כל קובצי הזיכרון בכל הריפוים בבת אחת.
- סנכרון transcripts בין מחשבים.

## שער קידום
רכיב עובר לשלב הבא רק אם:
1. קיימת בעיה מדידה.
2. קיימות לפחות שתי ראיות או אירוע חמור אחד.
3. נבדקה חלופה פשוטה יותר.
4. הוגדר מדד הצלחה.
5. הוגדר rollback.
6. יניב אישר שינוי בריפו היעד.
