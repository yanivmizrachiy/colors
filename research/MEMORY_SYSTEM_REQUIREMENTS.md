# דרישות מערכת הזיכרון האישית של קלוטקורד

עודכן: 2026-07-15

> מסמך מחקר ותכנון בלבד. מקור הכללים המחייב היחיד הוא `CLAUDE.md`.

## מטרת המערכת
לאפשר ל־Claude ללמוד לאורך זמן את הדרישות, ההעדפות, ההחלטות והחוזים של יניב, ולהפעיל אותם בעבודה עתידית בלי צורך להסביר הכול מחדש—אך גם בלי ליצור כפילויות, סתירות, state מיושן, חשיפת מידע רגיש או context קבוע גדול.

## הבעיות שנמצאו במחקר הריפוים
1. כללים קבועים, מצב נוכחי, רשימת “בוצע” והיסטוריה נשמרים לעיתים באותו קובץ.
2. אותה דרישה מופיעה כמה פעמים בניסוחים כמעט זהים.
3. מסמכים שונים מציגים ריפו או תיקייה שונים כמקור האמת.
4. שם הריפו, זהות המוצר והקוד הפעיל אינם תמיד תואמים.
5. snapshot של חומרה, גרסה, אחוז ביצוע או רשימת נתונים נשמר בתוך דף כללים ומתיישן.
6. פרויקטים מסוימים מפורטים מאוד אך טוענים כמה מסמכים בכל session.
7. פרויקטים אחרים כוללים קוד פעיל אך כמעט ללא זיכרון מוצר.
8. דרישות חשובות נשמרות בתוך commits, שיחות או קוד בלבד.
9. דוחות AI או מסמכי status מוצגים בטעות כמקור אמת למרות שאינם ראיה חיה.
10. קיימים עותקים, workspaces, child projects וריפו legacy ללא יחס קנוני אחיד.
11. מידע אישי או רגיש נשמר לעיתים בתוך Git ציבורי או בתוך דף כללים.
12. אחוזי “מוכנות” או “בוצע” אינם תמיד קשורים לבדיקה, commit או runtime אמיתי.

## סוגי מידע שחייבים להיות נפרדים
### 1. כלל קבוע
גבול או דרך עבודה שחלים כמעט בכל משימה בפרויקט.

### 2. העדפת יניב
העדפת UX, ניסוח, סגנון, דיוק, פדגוגיה או תהליך שניתנה במפורש על ידי יניב.

### 3. חוזה מוצר
התנהגות שאסור לשבור: מקור נתונים, הפרדת תפקידים, RTL, A4, fallback, הרשאות או נתיב פריסה.

### 4. החלטה ארכיטקטונית
בחירה עמידה עם סיבה, חלופות ותאריך.

### 5. מצב פעיל
מה קיים עכשיו, מה חסום ומה הצעד הבא. חייב להיות קצר, מתוארך וניתן לאימות מחדש.

### 6. משימה פתוחה
פעולה קונקרטית שעדיין לא הושלמה; אינה כלל קבוע.

### 7. ראיה
קובץ מקור, commit, test, screenshot, runtime output או מקור רשמי שתומכים בטענה.

### 8. היסטוריה
מה קרה בעבר. אינה נטענת בכל session ואינה גוברת על המצב או הדרישה הנוכחיים.

### 9. מידע רגיש
Secrets, פרטי תלמידים, פרטי קשר, כספים, runtime אישי, tokens ותמלילים. אינו נכנס ל־Git ציבורי או לזיכרון קבוע.

## מבנה היעד לכל פרויקט

- README למשתמש.

### Tier 2 — מוצר פעיל בינוני

- מסמך דרישות מוצר אחד או data/schema קנוניים.
- tests/guards לחוזים דטרמיניסטיים.

### Tier 3 — מוצר גדול או רגיש
- `CLAUDE.md` כשער קצר.
- state חי נפרד.
- מפת מקורות אמת לפי concern.
- evidence/canonical/reports מופרדים.
- rules לפי path או Skills לפי צורך.
- permissions/read-only defaults ואימות privacy.
- audit תקופתי של memory, tools ו־state.

## תהליך לימוד דרישה חדשה
1. Claude מזהה שהמשתמש נתן דרישה שעשויה להיות עמידה.
2. הוא מסווג אותה: כלל, העדפה, חוזה מוצר, החלטה, state או משימה.
3. הוא מחפש דרישה קיימת זהה או סותרת.
4. הוא מציג ליניב:
   - הניסוח הקנוני המוצע;
   - מה היא מחליפה או מעדכנת;
   - היכן היא תישמר;
   - אילו פרויקטים או נתיבים היא משפיעה עליהם.
5. רק לאחר אישור, הדרישה נכתבת במקום אחד.
6. אם היא דטרמיניסטית וחשובה, נבחן test/guard/permission/Hook במקום להסתפק בטקסט.
7. אם היא workflow חוזר ורב־שלבי, נבחן Skill לאחר שלוש חזרות או השפעה גבוהה.
8. מתעדים מקור, תאריך, scope ורמת ודאות.

## שינוי או ביטול דרישה
- ההוראה האחרונה של יניב אינה מתווספת אוטומטית לצד הישנה; היא מעדכנת או מחליפה אותה.
- Git שומר היסטוריה, ולכן אין צורך ביומן שינויים בתוך דף הכללים.
- דרישה מבוטלת מסומנת ב־decision/history רק כאשר יש ערך להבנת העבר.
- כל reference, test ו־Skill שתלויים בדרישה עוברים עדכון באותו שינוי.

## בדיקות סתירה לפני עבודה
לפני שינוי משמעותי המערכת בודקת:
- האם repo/branch/path תואמים לזהות הפרויקט.
- האם יותר ממסמך אחד טוען שהוא מקור אמת לאותו concern.
- האם state מתוארך אך יש commits חדשים יותר.
- האם README, rules והקוד מתארים מוצרים שונים.
- האם ריפו מוגדר canonical, child, donor, workspace, legacy או retired.
- האם קיימת דרישה חדשה שסותרת test/guard קיים.
- האם מידע רגיש נמצא במסמך ציבורי.

## מודל יחסים בין ריפוים
לכל ריפו יוגדר תפקיד אחד או יותר:
- `canonical-product` — המוצר הפעיל.
- `child-product` — מוצר ייעודי עם נתונים ו־URL משלו.
- `shared-infrastructure` — שירות או ספרייה שמשמשים כמה מוצרים.
- `control-plane` — תזמור/טריגרים, לא runtime המוצר.
- `runtime-agent` — ביצוע במכשיר או סביבה מסוימת.
- `source-library` — חומרי מקור בלבד.
- `donor-reference` — קריאה בלבד לצורך למידה.
- `workspace-export` — עותק עבודה זמני מכלי חיצוני.
- `legacy-pointer` — היסטוריה בלבד עם הפניה למקור הקנוני.
- `retired` — אסור להמשיך לפתח.

## טעינת context
- מידע שחייב להיות בכל משימה בלבד נכנס ל־`CLAUDE.md`.
- state נטען בתחילת משימה כאשר הוא נדרש.
- דרישות תחומיות נטענות לפי path או לפי סוג המשימה.
- evidence, reports, changelog והיסטוריה נקראים רק בעת חקירה.
- Skills נטענים/מופעלים לפי workflow, לא כהעתק נוסף של כללים.
- אין `@import` למסמכים גדולים רק כדי לארגן אותם—הם עדיין צורכים context.

## אמינות ו־provenance
לכל פריט זיכרון עמיד רצוי לשמור:
- `id` יציב.
- סוג המידע.
- ניסוח קנוני.
- scope: גלובלי / פרויקט / path / feature.
- מקור: יניב / קובץ / commit / API / test / runtime.
- תאריך יצירה ועדכון.
- סטטוס: proposed / approved / verified / superseded / revoked.
- רמת רגישות.
- references לקוד, tests או docs.
- תאריך review הבא כאשר המידע עשוי להתיישן.

## פרטיות
- אין להכניס שמות תלמידים, ציונים, נוכחות, טלפונים, מיילים אישיים, מידע פיננסי או runtime אישי לזיכרון ציבורי.
- מידע רגיש נשמר במערכת פרטית, DB, config מקומי או vault מתאים.
- Claude מקבל רק את המידע המינימלי הדרוש למשימה.
- דוח מסונן או fingerprint אינו מחליף בדיקת צורך ואישור.

## ביקורת תקופתית
אחת לחודש או לאחר שינוי גדול:
1. בדיקת גודל `CLAUDE.md` ומה נטען בפועל.
2. איתור דרישות כפולות או סותרות.
3. איתור state ישן ביחס ל־commits/runtime.
4. איתור Skills/Plugins/MCP שלא היו בשימוש.
5. איתור מידע רגיש או URLs/נתיבים אישיים.
6. העברת כללים דטרמיניסטיים ל־tests/guards.
7. סימון מסמכים כ־legacy או מחיקתם לאחר אימות references.
8. דיווח מה השתנה ומה דורש החלטת יניב.
