# דרישות מערכת הזיכרון האישית של קלוטקורד

עודכן: 2026-07-15

> מסמך מחקר ותכנון בלבד. מקור הכללים המחייב היחיד הוא `CLAUDE.md`.

## המטרה
לאפשר ל־Claude ללמוד לאורך זמן את הדרישות, ההעדפות, ההחלטות וחוזי המוצר של יניב, ולהפעיל אותם בעבודה עתידית בלי צורך להסביר הכול מחדש—בלי כפילויות, סתירות, state מיושן, מידע רגיש או context קבוע גדול.

## הבעיות שנמצאו
- כללים, state, היסטוריה ורשימת “בוצע” נשמרים לעיתים באותו קובץ.
- אותה דרישה מופיעה בכמה ניסוחים.
- מסמכים שונים מציגים מקורות אמת שונים.
- snapshots של חומרה, גרסה או אחוז מוכנות נשמרים בדף כללים ומתיישנים.
- חלק מהפרויקטים מפורטים מדי; אחרים כמעט ללא זיכרון מוצר.
- דרישות חשובות נשמרות בתוך commits, שיחות או קוד בלבד.
- status או דוח AI מוצגים לעיתים כראיה חיה למרות שאינם מאומתים.
- מידע אישי או רגיש עלול להיכנס ל־Git ציבורי.

## סוגי מידע שחייבים להיות נפרדים
1. כלל קבוע.
2. העדפה של יניב.
3. חוזה מוצר.
4. החלטה ארכיטקטונית.
5. מצב פעיל.
6. משימה פתוחה.
7. ראיה.
8. היסטוריה.
9. מידע רגיש שאסור לשמור בזיכרון ציבורי.

## תהליך קליטת דרישה
1. Claude מזהה מועמד לדרישה עמידה.
2. מסווג את הדרישה.
3. מחפש כפילות, סתירה ו־test/guard קיים.
4. מציג ליניב ניסוח קנוני, scope, מיקום ומה יוחלף.
5. כותב רק לאחר אישור.
6. מעדכן את הדרישה הקיימת במקום להוסיף גרסה מתחרה.
7. מציע test, verifier, Rule, Permission, Skill או Hook כאשר טקסט בלבד אינו מספיק.

## מבנה רשומה מומלץ
- `id` יציב.
- סוג.
- ניסוח קנוני.
- scope.
- מקור ותאריך.
- סטטוס: proposed / approved / verified / superseded / revoked.
- רמת רגישות.
- references לקוד, tests ו־docs.
- תאריך review כאשר המידע עשוי להתיישן.

## טעינת context
- מידע שחייב להיות בכל משימה בלבד נכנס ל־`CLAUDE.md`.
- state נטען רק כשנדרש.
- דרישות תחומיות נטענות לפי path או סוג משימה.
- evidence, reports והיסטוריה נקראים רק בעת חקירה.
- Skills מפעילים workflow; הם אינם עותק נוסף של כללי הפרויקט.

## פרטיות
- אין לשמור שמות תלמידים, ציונים, טלפונים, מיילים אישיים, מידע פיננסי, tokens או transcripts בזיכרון ציבורי.
- מידע רגיש נשמר במערכת פרטית, DB, config מקומי או vault מתאים.
- Claude מקבל רק את המידע המינימלי הדרוש למשימה.

## ביקורת תקופתית
אחת לחודש או לאחר שינוי גדול:
- בדיקת גודל `CLAUDE.md` ומה נטען בפועל.
- איתור דרישות כפולות או סותרות.
- איתור state ישן ביחס לקוד ול־runtime.
- איתור Skills/Plugins/MCP שלא היו בשימוש.
- איתור מידע רגיש.
- העברת כללים דטרמיניסטיים ל־tests/guards.
- סימון מסמכים כ־legacy או מחיקתם לאחר אימות references.

## מדדי הצלחה
- פחות דרישות שיניב מסביר מחדש.
- אפס דרישות קריטיות שנשכחו בפיילוט.

- פחות context פתיחה.
- פחות סבבי תיקון עקב אי־הבנת העדפות.

- קלות עדכון, ביטול ו־rollback של דרישה.
