# קלוטקורד — דף הכללים היחיד

> זהו מקור הכללים המחייב היחיד במיזם. כל קובץ אחר הוא מצב, מחקר, ראיה, תבנית או תוכנית — ואינו מוסיף כללים מחייבים.

## 1. מטרת המיזם
לבנות עבור יניב מערכת עבודה אמיתית, מהירה, מדויקת ובטוחה עם Claude Code, המותאמת לפרויקטים ולדרך העבודה שלו — לא דמו, לא תרגיל ולא אוסף טיפים כללי.

## 2. אמת, היקף ומקורות
- עובדים על פרויקטים אמיתיים ונתונים אמיתיים בלבד; אין placeholders, demo או טענות הצלחה מומצאות.
- לפני פעולה מאמתים repository, branch, remote, working tree והיקף המשימה.
- `CURRENT_STATE.md` הוא מקור המצב החי; `docs/ROADMAP.md` הוא מפת השלבים; Issues הם checklist תפעולי.
- מסמכי `docs/` ו־`research/` הם חומרי מחקר והחלטה בלבד.
- מפרידים תמיד בין עובדה, הסקה, השערה, המלצה, מועמד שינוי ומצב פעיל.
- עובדה חיצונית או עדכנית חייבת מקור ותאריך; אין לנחש גרסאות, יכולות או התנהגות של כלי.

## 3. גבולות שינוי
- בריפו המרכזי מותר לחקור, לתעד, לסנכרן מצב ולבנות תוכניות ופיילוטים.
- שינוי בריפו יעד נעשה רק בהיקף שאושר, בענף ייעודי וב־PR, עם rollback ברור.
- אין merge, deploy, שינוי DB/auth/secrets, מחיקה רחבה או שינוי קוד מוצר ללא שער אישור מתאים.
- אין push ישיר ל־`main`, force-push, `reset --hard`, `git clean` או שכתוב היסטוריה ללא הוראה מפורשת.
- עבודה מקומית קיימת שאינה קשורה למשימה אינה נמחקת או נדרסת.

## 4. אבטחה ופרטיות
- אין לקרוא, להדפיס, לשמור או להעלות secrets, credentials, tokens, cookies, passwords, API keys או ערכי `.env`.
- אין לשמור תמלילי Claude Code גולמיים, נתוני תלמידים, ציונים, שמות תלמידים, browser profiles או OAuth data.
- הריפו ציבורי כל עוד לא הוכח אחרת; כל חומר בו חייב להיות כללי ולא־רגיש.
- כלי, Skill, Plugin, Hook, MCP או סקריפט חיצוני נבדק לפני התקנה: מקור, קוד, הרשאות, פעולות צד, עדכונים ויכולת הסרה.
- אין להריץ התקנה או סקריפט מרוחק לפי שם שהמודל הציע בלי אימות מול מקור רשמי או repository אמיתי.

## 5. דרך העבודה בכל משימה
1. מגדירים תוצאה אחת ברורה, היקף, אזורים מוגנים, מדד הצלחה ובדיקות.
2. חוקרים לפני קוד כאשר הגישה אינה ברורה, השינוי רב־קבצים או הסיכון בינוני ומעלה.
3. במשימה קטנה וברורה מדלגים על תכנון כבד ומבצעים שינוי ממוקד.
4. מחפשים לפני יצירת קובץ חדש ולפני קריאת מסמך גדול.
5. משנים את המינימום הדרוש; אין refactor או שיפור צדדי ללא צורך.
6. נותנים ל־Claude דרך לאמת את עבודתו: test, typecheck, build, preview, screenshot, environment או live לפי הסיכון.
7. קוראים את כל ה־diff לפני commit.
8. פותחים PR ומדווחים מה בוצע, מה עבר, מה אומת חי ומה עדיין לא אומת.

## 6. Git, בדיקות ו־release readiness
- GitHub הוא מקור האמת לקוד ולשינויים.
- שינוי משמעותי מתבצע בענף וב־PR; commits מחולקים ליחידות הגיוניות והפיכות.
- PASS של lint/typecheck/build אינו שווה READY.
- בדיווח מפרידים בין: implemented, tests passed, preview/environment verified, live verified, release ready.
- שינוי UI, מצגת, PDF, RTL או חומר לימודי מחייב בדיקה חזותית ולא רק build.
- שינוי תלוי שירות חיצוני מחייב בדיקת environment/live נפרדת.

## 7. Context, טוקנים ומודלים
- משימה אחת משמעותית לכל session; מעבר למשימה לא קשורה מחייב `/rename` ואז `/clear`.
- משתמשים ב־`/compact` עם הוראה ממוקדת רק כאשר ממשיכים את אותה משימה.
- לאחר שני ניסיונות תיקון כושלים באותו כיוון, עוצרים, מסכמים ופותחים session נקי.
- Sonnet הוא ברירת המחדל לעבודה שוטפת; Opus/effort גבוה שמורים לתכנון מורכב, root cause, אבטחה או סיכון גבוה.
- לא מפעילים Agent Teams כברירת מחדל; כל teammate צורך context נפרד.
- Subagent מתאים לחקירה מבודדת ורועשת; השיחה הראשית מתאימה לשלבים שחולקים context ולשינויים ממוקדים.
- מודדים עם `/usage`, `/context` וזמן אמיתי לפני טענה לחיסכון.

## 8. Skills, Hooks, Plugins, MCP ואוטומציות
- `CLAUDE.md` מכיל רק מידע שחייב להיות בכל session.
- Skill נבנה או מותקן רק לתהליך חוזר ורב־שלבי שהופיע לפחות שלוש פעמים או שהשפעתו גבוהה במיוחד.
- Skill עם פעולות צד מסוכנות יהיה ידני (`disable-model-invocation: true`) ועם כלים מוגבלים.
- Hook משמש רק לפעולה שחייבת לקרות תמיד או לחסימה דטרמיניסטית; לא להפעלת build כבד אחרי כל עריכה.
- Permissions ו־sandbox קודמים ל־Hook כאשר הם יכולים לאכוף את הגבול בפשטות.
- Plugin מועדף ממקור רשמי; Plugin קהילתי דורש review של הקוד וה־commit שאליו הוא pinned.
- MCP נשמר רק אם הוא מחליף עבודה ידנית חוזרת; הרשאות כתיבה מצומצמות והשרת מאושר במפורש.
- LSP/code-intelligence הוא מועמד בעדיפות גבוהה לפרויקטי TypeScript, לאחר בדיקת התאמה בפיילוט.
- Scheduled tasks, non-interactive mode ואוטומציה רציפה יופעלו רק עם תנאי עצירה, לוג מצומצם וגבולות הרשאה.
- לכל הרחבה מתעדים: הבעיה, תדירות, חלופה פשוטה יותר, עלות context, סיכון, תחזוקה, מדד הצלחה ו־rollback.

## 9. ניהול ידע
- כלל מחייב נכתב רק בקובץ זה; אין לשכפל אותו במסמך אחר.
- מידע זמני נשמר רק ב־`CURRENT_STATE.md`.
- החלטה עמידה נשמרת ב־`DECISIONS.md` כהיסטוריית החלטות, לא כהוראות עבודה.
- מחקר רשמי וקהילתי נשמר תחת `docs/` או `research/` עם מקורות ותאריך.
- קובץ מיושן, כפול או שאינו משרת את מטרת קלוטקורד נמחק לאחר אימות שאין אליו תלות פעילה.
- אין להעמיס היסטוריה, מפת קבצים או מדריכים ארוכים על דף הכללים.

## 10. הגדרת השלמה
עבודה נחשבת שלמה רק כאשר:
- התוצאה והיקף השינוי ברורים.
- הבדיקות המתאימות בוצעו והראיות נשמרו.
- אין מידע רגיש או שינוי צדדי.
- branch, commit, PR, מצב merge ו־rollback דווחו.
- `CURRENT_STATE.md` וה־Issue הפעיל תואמים למציאות.
- הצעד הבא ברור ואינו מוצג כאילו כבר בוצע.
